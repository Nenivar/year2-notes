Shortest Paths Revisited

Negative Weights

Bellman-Ford's algorithm solves the single source shortest paths problem in a weighted, directed graph.

  • It finds the shortest path from a given source vertex to every other vertex.
  • The weights are allowed to be positive or negative.
  • The graph is started as an adjacency list

Reminder: Dijkstra's algorithm solves this problem in O(∣V∣+∣E∣)log⁡∣V∣)O(\vert V\vert+\vert E\vert)\log\vert V\vert)O(∣V∣+∣E∣)log∣V∣) time when the edges have non-negative weights

Negative Weight Cycles

A negative weight cycle is a path from vertex vvv back to vvv s.t. the sum of the edge weights is negative.

If there is a path from sss to ttt which includes a negative weight cycle, there is no shortest path from sss to ttt.

Most of the Bellman-Ford Algorithm

It repeatedly asks, for each edge (u,v)(u,v)(u,v) 'can I find a shorter route to vvv if I go via uuu?'

Claim:

When the MOSTOFBELLMAN_FORD algorithm terminates, for each vertex v, dist(v) is the length of the shortest path between s and v.

Proof Idea

Imaginary Alg.

Imagine a different algorithm where in each iteration you relax only one edge.

Further, imagine that the edge you relax in iteration iii is the ithi^{th}ith edge in a shortest path from sss to some vertex ttt.

When this algorithm terminates, dist(t)dist(t)dist(t) is the length of the shortest path from sss to ttt.

MOSTOFBELLMAN_FORD

You relax every edge in each iteration.

Consider the same shortest path from sss to ttt.

At some point in iteration iii, you relax the ithi^{th}ith edge in the shortest path from sss to ttt.

After enough iterations, dist(t)dist(t)dist(t) is the length of a path from sss to ttt, which is at least as short as the shortest path.

We now prove that if are no negative cycles in the graph, the shortest path between sss and ttt contains at most ∣V∣\vert V\vert∣V∣ edges (and therefore ∣V∣\vert V\vert∣V∣) iterations will be enough.

Consider a path between sss and ttt with a cycle:

As there are no negative weight cycles, deleting this cycle from the path cannot increase its length ∴\therefore∴ there is a shortest path between sss and ttt containing no cycles A path with no cycles enters each vertex at most once, so contains at most ∣V∣\vert V\vert∣V∣ edges

Full Algorithm

def BELLMAN_FORD(s):
    for all v, set dist(v) = infty
    set dist(s) = 0
    for i in range(0, len(V)):
        for every edge (u,v) in E:
            if dist(v) > dist(u) + weight(u,v): # relaxing edge (u,v)
                dist(v) = dist(u) + weight(u,v)
    
    for every edge (u,v) in E:
        if dist(v) > dist(u) + weight(u,v):
            print('neg. weight cycle found')

We added a final check which determines whether another iteration would find an even shorter path from some sss to some vvv.

The algorithm still works when there is no negative weight cycle:

After the first ∣V∣\vert V\vert∣V∣ iterations (which haven't changed), for each vertex vvv, dist(v)dist(v)dist(v) is the length of the shortest path between sss and vvv.

If the final check outputs 'neg. weight cycle found', then there is a path from sss to some vvv (via uuu) with length:

dist(u)+weight(u,v)<dist(v)dist(u) + weight(u,v) < dist(v) dist(u)+weight(u,v)<dist(v)

i.e. a path which is shorter than the shortest path - contradiction!

All-Pairs

What do we do to find the shortest paths from every vertex to every other vertex?

We could reuse:

  • non-negative edge weights:
    • run DIJKSTRA's ∣V∣\vert V\vert∣V∣ times; once with each vertex as the source
    • this takes O(∣V∣∣E∣log⁡∣V∣)O(\vert V\vert\vert E\vert \log\vert V\vert)O(∣V∣∣E∣log∣V∣) time
  • negative edge weights:
    • run BELLMAN_FORD ∣V∣\vert V\vert∣V∣ times; once for each vertex
    • this takes O(∣V∣2∣E∣)O(\vert V\vert^2\vert E\vert)O(∣V∣2∣E∣) time

The output contains the length of the shortest path between every pair of vertices.

There are ∣V∣⋅(∣V∣−1)\vert V\vert \cdot (\vert V\vert -1)∣V∣⋅(∣V∣−1) pairs of vertices, so we can't expect to do better than O(∣V∣2)O(\vert V\vert^2)O(∣V∣2) time.

If we imagine ∣E∣≈5∣V∣\vert E\vert \approx 5\vert V\vert∣E∣≈5∣V∣ (the graph is very spare), e.g. each vertex has an edge to about 555 other vertices, Dijkstra's takes O(∣V∣2log⁡∣V∣)O(\vert V\vert^2\log\vert V\vert)O(∣V∣2log∣V∣) time (a lot better than the graph being dense).

JOHNSON's Algorithm

This works for positive and negative edge weights, and uses both DIJKSTRA's and BELLMAN_FORD.

Time complexity:

O(∣V∣∣E∣log⁡∣V∣)O(\vert V\vert\vert E\vert \log\vert V\vert) O(∣V∣∣E∣log∣V∣)

The approach employed by JOHNSON's algorithm is to reweight the edges, so that the resulting graph has non-negative edge weights (and then repeatedly runs DIJKSTRA's algorithm).

First Attempt

One natural attempt is to increase every edge weight by the same amount.

Unforunately, both the shortest paths lengths and the routes might change.

Reweighting Based on Vertex Potential

To overcome this, we're going to reweight each edge differently - the new weight of each edge will depend on which vertices it connects.

h(v)h(v)h(v) is the potential of a vertex vvv.

where weight′(A,B)=weight(A,B)+h(A)−h(B)weight'(A,B)=weight(A,B) + h(A) - h(B)weight′(A,B)=weight(A,B)+h(A)−h(B)

  1. Add one additional vertex called sss to the original graph

  2. Add an edge (s,v)(s,v)(s,v) from sss to each other vertex v∈Vv\in Vv∈V; each of these edges has weight 000

  3. For each vvv, let δ(s,v)\delta(s,v)δ(s,v) denote the length of the shortest path from sss to vvv

  4. Consider any edge (u,v)∈E(u,v)\in E(u,v)∈E

    So the shortest path can't be longer
